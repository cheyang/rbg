# 通过 Warmup 加速服务就绪时间

## 概述
大型推理模型（如 DeepSeek R1 671B）的服务部署面临严峻的冷启动挑战：模型权重超过 600 GB、推理引擎镜像数十 GB、GPU kernel 首次编译需要数分钟。如果直接部署推理服务，每个 Pod 都需要从零开始拉取镜像、下载模型、编译 kernel，导致服务就绪时间过长。

`RoleBasedGroupWarmup` 是一个独立的 CRD，允许你在部署推理服务**之前**，在目标节点上预先完成镜像拉取、模型下载、kernel 编译等准备工作。当推理服务启动时，这些资源已在节点上就绪，大幅缩短服务就绪时间。

---

## 前提条件
+ Kubernetes 集群版本 >= 1.24
+ 已安装 RBG Controller（参考 [安装指南](../install.md)）
+ 集群中包含 GPU 节点（DeepSeek R1 671B 需要至少 8 张 GPU 进行张量并行）

> **说明**：本文以 DeepSeek R1 671B（FP8）为例演示 Warmup 的完整流程。Warmup 机制适用于任意推理引擎和模型。
>

---

## 背景：DeepSeek R1 的冷启动分析
DeepSeek R1 是一个 671B 参数的大型语言模型，使用 FP8 量化后权重约 671 GB。在 8 卡 GPU 节点上部署该模型，Pod 从创建到就绪需要经历以下阶段：

```plain
┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  镜像拉取     │─→│  模型下载     │─→│  模型加载     │─→│  Kernel 编译  │─→│  引擎就绪     │
│  ~3 分钟      │  │  ~10-30 分钟  │  │  ~3-5 分钟    │  │  ~2-5 分钟    │  │  接收流量      │
│  (27 GB 镜像) │  │  (671 GB 权重)│  │  (GPU 加载)   │  │  (DeepGEMM)  │  │              │
└──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘  └──────────────┘
```

总冷启动时间可达 **20-40 分钟**，其中大部分可以通过 Warmup 提前完成：

| 阶段 | 冷启动耗时 | Warmup 后耗时 | 说明 |
| --- | --- | --- | --- |
| 镜像拉取 | ~3 分钟 | 跳过（已在节点） | 镜像预拉取 |
| 模型下载 | ~10-30 分钟 | 跳过（已在 hostpath） | 模型预下载到节点本地 |
| DeepGEMM Kernel 编译 | ~2-5 分钟 | 跳过（预编译文件已在 hostpath） | 提前编译并缓存到节点 |
| 模型加载到 GPU | ~3-5 分钟 | ~3-5 分钟 | 无法跳过，必须运行时加载 |


通过 Warmup，服务就绪时间从 **20-40 分钟缩短到 ~5 分钟**。

> **说明**：以上耗时数据为经验估算值，实际耗时受网络带宽、存储 IOPS、GPU 型号等因素影响。建议在自己的集群上实测各阶段耗时，以获得准确的基准数据。

## Warmup CRD 配置
`RoleBasedGroupWarmup` 支持两种目标模式：

| 模式 | 字段 | 说明 |
| --- | --- | --- |
| 指定节点 | `spec.targetNodes` | 通过 `nodeSelector` 或 `nodeNames` 显式指定目标节点 |
| 引用 RBG | `spec.targetRoleBasedGroup` | 从已有的 RBG 资源中自动发现节点 |


本文使用 `targetNodes` 模式，将预热动作部署到指定的 GPU 节点上。

### 参数说明
| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `spec.targetNodes.nodeSelector` | map[string]string | 否（与 nodeNames 二选一） | 节点标签选择器，匹配所有符合条件的节点 |
| `spec.targetNodes.nodeNames` | []string | 否（与 nodeSelector 二选一） | 显式指定节点名称列表 |
| `spec.targetNodes.imagePreload.items` | []string | 否 | 需要预拉取的容器镜像列表 |
| `spec.targetNodes.imagePreload.pullSecrets` | []object | 否 | 镜像拉取凭证 |
| `spec.targetNodes.customizedAction.containers` | []Container | 否 | 自定义预热容器，用于模型下载、kernel 编译等 |
| `spec.targetNodes.customizedAction.volumes` | []Volume | 否 | 自定义预热容器使用的 Volume |


> **说明**：`imagePreload` 和 `customizedAction` 可以同时配置，也可以单独使用。
>

## 场景一：预热容器镜像
推理引擎镜像（如 SGLang）解压后通常有 20-30 GB。首次部署时，每个节点都需要从镜像仓库拉取，耗时数分钟。通过 `imagePreload`，可以让目标节点提前拉取并缓存镜像。

```yaml
apiVersion: workloads.x-k8s.io/v1alpha2
kind: RoleBasedGroupWarmup
metadata:
  name: deepseek-r1-image-warmup
spec:
  targetNodes:
    nodeSelector:
      gpu-type: h100
      model-target: deepseek-r1
    imagePreload:
      items:
        - "lmsysorg/sglang:v0.5.9"
```

### 工作原理
Controller 在每个匹配的节点上创建一个 Warmup Pod，该 Pod 包含一个镜像预热容器：

+ 使用指定的镜像（`lmsysorg/sglang:v0.5.9`）
+ 启动命令为 `exit 0`，即拉取镜像后立即退出
+ kubelet 在拉取过程中已将镜像缓存到节点本地存储

当后续推理服务 Pod 调度到同一节点并使用相同镜像时，kubelet 直接使用本地缓存，跳过网络拉取。

## 场景二：预热模型文件
DeepSeek R1 的模型权重超过 600 GB，从远程存储（如 HuggingFace、S3）下载是最大的冷启动瓶颈。通过 `customizedAction`，可以在预热容器中提前将模型下载到节点的 hostpath 目录，推理服务启动时直接挂载该目录。

```yaml
apiVersion: workloads.x-k8s.io/v1alpha2
kind: RoleBasedGroupWarmup
metadata:
  name: deepseek-r1-model-warmup
spec:
  targetNodes:
    nodeSelector:
      gpu-type: h100
      model-target: deepseek-r1
    imagePreload:
      items:
        - "lmsysorg/sglang:v0.5.9"
    customizedAction:
      volumes:
        - name: model-cache
          hostPath:
            path: /mnt/models/deepseek-r1
            type: DirectoryOrCreate
      containers:
        - name: download-model
          image: lmsysorg/sglang:v0.5.9
          command:
            - bash
            - -c
            - |
              echo "开始下载 DeepSeek R1 模型..."
              python3 -c "
              from huggingface_hub import snapshot_download
              snapshot_download(
                repo_id='deepseek-ai/DeepSeek-R1',
                local_dir='/mnt/models/deepseek-r1',
                local_dir_use_symlinks=False
              )
              print('模型下载完成')
              "
          volumeMounts:
            - name: model-cache
              mountPath: /mnt/models/deepseek-r1
```

### 工作原理
1. Warmup Pod 调度到目标节点，挂载 hostpath `/mnt/models/deepseek-r1`
2. 预热容器从 HuggingFace 下载模型权重到该目录
3. 下载完成后，模型文件持久化在节点本地磁盘
4. 后续推理服务 Pod 挂载同一 hostpath，直接从本地读取模型，跳过下载

> **说明**：使用 hostpath 可以确保模型文件在 Pod 生命周期结束后仍然保留在节点上。推理服务 Pod 只需挂载相同的 hostpath 路径即可复用。
>

## 场景三：自定义预热动作 — DeepGEMM Kernel 编译
DeepSeek R1 使用 DeepGEMM 进行高效的矩阵运算。DeepGEMM 的 CUDA kernel 在首次运行时需要 JIT（Just-In-Time）编译，编译过程需要 2-5 分钟，且编译产物可以缓存在本地磁盘上。

通过自定义预热容器，可以在推理服务部署**之前**提前编译 DeepGEMM kernel，并将预编译文件保存到 hostpath。推理服务启动时挂载同一 hostpath，直接复用预编译结果，跳过 JIT 编译。

```yaml
apiVersion: workloads.x-k8s.io/v1alpha2
kind: RoleBasedGroupWarmup
metadata:
  name: deepseek-r1-full-warmup
spec:
  targetNodes:
    nodeSelector:
      gpu-type: h100
      model-target: deepseek-r1
    imagePreload:
      items:
        - "lmsysorg/sglang:v0.5.9"
    customizedAction:
      volumes:
        - name: model-cache
          hostPath:
            path: /mnt/models/deepseek-r1
            type: DirectoryOrCreate
        - name: deepgemm-cache
          hostPath:
            path: /mnt/cache/deepgemm
            type: DirectoryOrCreate
      containers:
        # 容器 1：下载模型权重
        - name: download-model
          image: lmsysorg/sglang:v0.5.9
          command:
            - bash
            - -c
            - |
              if [ -f /mnt/models/deepseek-r1/config.json ]; then
                echo "模型已存在，跳过下载"
                exit 0
              fi
              echo "开始下载 DeepSeek R1 模型..."
              python3 -c "
              from huggingface_hub import snapshot_download
              snapshot_download(
                repo_id='deepseek-ai/DeepSeek-R1',
                local_dir='/mnt/models/deepseek-r1',
                local_dir_use_symlinks=False
              )
              print('模型下载完成')
              "
          volumeMounts:
            - name: model-cache
              mountPath: /mnt/models/deepseek-r1

        # 容器 2：预编译 DeepGEMM kernel
        - name: compile-deepgemm
          image: lmsysorg/sglang:v0.5.9
          command:
            - bash
            - -c
            - |
              echo "开始预编译 DeepGEMM kernel..."
              export DEEPGEMM_JIT_DIR=/mnt/cache/deepgemm/jit_cache
              mkdir -p $DEEPGEMM_JIT_DIR
              python3 -c "
              import deep_gemm
              deep_gemm.compile_all()
              print('DeepGEMM kernel 预编译完成')
              "
          env:
            - name: DEEPGEMM_JIT_DIR
              value: /mnt/cache/deepgemm/jit_cache
          volumeMounts:
            - name: model-cache
              mountPath: /mnt/models/deepseek-r1
              readOnly: true
            - name: deepgemm-cache
              mountPath: /mnt/cache/deepgemm
          resources:
            limits:
              nvidia.com/gpu: "1"
```

> **重要：容器并行执行**。`customizedAction` 中的多个容器在同一个 Pod 内作为普通容器（非 initContainers）运行，因此它们会**并行启动**。如果容器之间存在依赖关系（如 `compile-deepgemm` 需要 `download-model` 先完成下载模型文件），你需要在容器的启动脚本中自行处理等待逻辑。上面示例中 `download-model` 容器包含了文件存在性检查（`if [ -f /mnt/models/deepseek-r1/config.json ]`），`compile-deepgemm` 容器也应添加类似的等待逻辑，例如：
>
> ```bash
> # 等待模型下载完成
> while [ ! -f /mnt/models/deepseek-r1/config.json ]; do
>   echo "等待模型下载完成..."
>   sleep 10
> done
> ```

### 工作原理
```plain
预热阶段（推理服务部署前）
┌─────────────────────────────────────────────────────────────┐
│  Warmup Pod (调度到目标 GPU 节点)                             │
│                                                             │
│  ┌──────────────────────┐    ┌──────────────────────┐      │
│  │ download-model        │    │ compile-deepgemm      │      │
│  │                       │    │                       │      │
│  │ HuggingFace →         │    │ JIT 编译 →            │      │
│  │ /mnt/models/          │    │ /mnt/cache/deepgemm/  │      │
│  │ deepseek-r1/          │    │ jit_cache/            │      │
│  │                       │    │                       │      │
│  │ 挂载: hostpath        │    │ 挂载: hostpath        │      │
│  │ /mnt/models/          │    │ /mnt/cache/deepgemm/  │      │
│  │ deepseek-r1           │    │                       │      │
│  └──────────────────────┘    └──────────────────────┘      │
│                    │                      │                  │
│                    ▼                      ▼                  │
│            节点本地磁盘 (hostpath)                            │
│            ┌────────────────────────────────────┐           │
│            │ /mnt/models/deepseek-r1/  (671 GB)  │           │
│            │ /mnt/cache/deepgemm/      (预编译)   │           │
│            └────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────────┘

推理服务启动阶段
┌─────────────────────────────────────────────────────────────┐
│  推理服务 Pod (调度到同一节点，挂载相同 hostpath)               │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐   │
│  │ SGLang Engine                                        │   │
│  │                                                      │   │
│  │ 模型: 直接从 /mnt/models/deepseek-r1/ 加载 (本地)     │   │
│  │ DeepGEMM: 直接从 /mnt/cache/deepgemm/ 复用 (已编译)   │   │
│  │                                                      │   │
│  │ 跳过: 镜像拉取 ✓  模型下载 ✓  Kernel 编译 ✓           │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

## 部署推理服务
Warmup 完成后，部署推理服务时需要确保两点：

1. **节点亲和性**：推理服务 Pod 调度到已预热的节点上
2. **挂载相同 hostpath**：复用预热阶段保存的模型文件和预编译文件

### 推理服务配置
以 DeepSeek R1 聚合部署 + TP=8 为例：

```yaml
apiVersion: workloads.x-k8s.io/v1alpha2
kind: RoleBasedGroup
metadata:
  name: deepseek-r1-inference
spec:
  roles:
    - name: backend
      replicas: 1
      leaderWorkerPattern:
        size: 8   # 1 Leader + 7 Workers = 8 GPU 张量并行
        template:
          spec:
            # 调度到已预热的节点
            affinity:
              nodeAffinity:
                requiredDuringSchedulingIgnoredDuringExecution:
                  nodeSelectorTerms:
                    - matchExpressions:
                        - key: gpu-type
                          operator: In
                          values: ["h100"]
                        - key: model-target
                          operator: In
                          values: ["deepseek-r1"]
            containers:
              - name: engine
                image: lmsysorg/sglang:v0.5.9
                command:
                  - python3
                  - -m
                  - sglang.launch_server
                  - --model-path
                  - /mnt/models/deepseek-r1
                  - --host
                  - "0.0.0.0"
                  - --port
                  - "8000"
                  - --tp-size
                  - "8"
                  - --trust-remote-code
                  - --quantization
                  - fp8
                  - --dist-init-addr
                  - $(RBG_LWP_LEADER_ADDRESS):6379
                  - --nnodes
                  - $(RBG_LWP_GROUP_SIZE)
                  - --node-rank
                  - $(RBG_LWP_WORKER_INDEX)
                ports:
                  - name: http
                    containerPort: 8000
                env:
                  # DeepGEMM 使用预热阶段编译的 kernel
                  - name: DEEPGEMM_JIT_DIR
                    value: /mnt/cache/deepgemm/jit_cache
                resources:
                  requests:
                    nvidia.com/gpu: "1"
                  limits:
                    nvidia.com/gpu: "1"
                volumeMounts:
                  - name: dshm
                    mountPath: /dev/shm
                  - name: model-cache
                    mountPath: /mnt/models/deepseek-r1
                    readOnly: true
                  - name: deepgemm-cache
                    mountPath: /mnt/cache/deepgemm
                    readOnly: true
            volumes:
              - name: dshm
                emptyDir:
                  medium: Memory
                  sizeLimit: 30Gi
              - name: model-cache
                hostPath:
                  path: /mnt/models/deepseek-r1
                  type: Directory
              - name: deepgemm-cache
                hostPath:
                  path: /mnt/cache/deepgemm
                  type: Directory
```

### 关键配置说明
| 配置项 | 作用 |
| --- | --- |
| `nodeAffinity` + `requiredDuringScheduling` | 确保 Pod 调度到已预热的节点（与 Warmup 的 `nodeSelector` 使用相同的标签） |
| `hostPath: /mnt/models/deepseek-r1` | 复用预热阶段下载的模型文件 |
| `hostPath: /mnt/cache/deepgemm` | 复用预热阶段编译的 DeepGEMM kernel |
| `--model-path /mnt/models/deepseek-r1` | 从本地 hostpath 加载模型，而非远程下载 |
| `DEEPGEMM_JIT_DIR` | 指向预编译缓存目录，跳过 JIT 编译 |


> **说明**：`nodeAffinity` 使用与 Warmup `nodeSelector` 相同的标签（`gpu-type: h100`, `model-target: deepseek-r1`），确保推理服务调度到已预热的节点。如果预热未完成就部署服务，Pod 仍会被调度（如果节点标签匹配），但无法享受预热带来的加速效果。
>

## 完整流程
将 Warmup 和推理服务组合使用的完整操作流程：

```plain
步骤 1: 为 GPU 节点添加标签
    $ kubectl label nodes <node-name> gpu-type=h100 model-target=deepseek-r1

步骤 2: 创建 Warmup 任务
    $ kubectl apply -f deepseek-r1-full-warmup.yaml

步骤 3: 等待 Warmup 完成
    $ kubectl get rbgwarmup deepseek-r1-full-warmup -w
    NAME                        PHASE       DESIRED   SUCCEEDED   FAILED
    deepseek-r1-full-warmup     Running     2         0           0
    deepseek-r1-full-warmup     Running     2         1           0
    deepseek-r1-full-warmup     Completed   2         2           0

步骤 4: 部署推理服务
    $ kubectl apply -f deepseek-r1-inference.yaml

步骤 5: 验证推理服务就绪
    $ kubectl get rbg deepseek-r1-inference
```

| 指标 | 无 Warmup | 有 Warmup |

| 镜像拉取 | ~3 分钟 | 跳过 |
| 模型下载 | ~10-30 分钟 | 跳过 |
| DeepGEMM 编译 | ~2-5 分钟 | 跳过 |
| 模型加载到 GPU | ~3-5 分钟 | ~3-5 分钟 |
| **总计** | **~20-40 分钟** | **~5 分钟** |

## Warmup 故障恢复

### 场景一：模型下载中断（部分文件已下载）

**现象**：模型文件有数百 GB，下载过程中网络中断，Warmup Pod 失败。hostpath 上留有不完整的模型文件。

**恢复步骤**：

```bash
# 步骤 1：检查 Warmup 状态
kubectl get rbgwarmup <warmup-name>

# 步骤 2：查看下载容器的日志，确认下载进度
kubectl logs <warmup-pod-name> -c customized-download-model

# 步骤 3：删除 Warmup 资源后重新创建
kubectl delete rbgwarmup <warmup-name>
kubectl apply -f <warmup-yaml>
```

> **说明**：如果下载脚本使用了 `huggingface_hub.snapshot_download`，该函数支持断点续传——已下载的文件不会重新下载。因此重新执行 Warmup 时，只会下载上次中断后未完成的部分。如果使用其他下载工具，请确保脚本具备断点续传能力，否则需要先清理 hostpath 上的不完整文件。

### 场景二：Warmup 成功但推理服务仍然冷启动

**现象**：Warmup 显示 Completed，但部署推理服务后，Pod 日志显示仍在从远程下载模型。

**排查步骤**：

```bash
# 步骤 1：确认推理 Pod 是否调度到了预热过的节点
kubectl get pod <inference-pod-name> -o wide  # 查看 NODE 列
kubectl get pods -l workloads.x-k8s.io/warmup-name=<warmup-name> -o wide  # 查看 Warmup 在哪些节点执行过

# 步骤 2：检查 hostpath 挂载路径是否一致
kubectl get pod <inference-pod-name> -o yaml | grep -A5 hostPath
# 对比 Warmup 中配置的 hostPath 路径

# 步骤 3：确认节点上的文件是否存在
kubectl exec <inference-pod-name> -- ls /mnt/models/deepseek-r1/
```

**常见原因**：
+ **节点不匹配**：推理 Pod 的 `nodeAffinity` 标签与 Warmup 的 `nodeSelector` 不一致，Pod 被调度到了未预热的节点
+ **路径不一致**：Warmup 下载到 `/mnt/models/deepseek-r1`，但推理服务挂载的是另一个路径
+ **模型参数不匹配**：推理引擎的 `--model-path` 指向了远程路径（如 HuggingFace repo ID）而非本地 hostpath

### 场景三：预热后升级镜像版本，需要重新预热吗？

**视情况而定**：

| 变更内容 | 是否需要重新 Warmup |
|---------|-------------------|
| 仅升级推理引擎镜像版本 | 需要重新预热**镜像**（`imagePreload`），模型文件不需要 |
| 切换模型（如从 Qwen 切到 DeepSeek） | 需要重新预热**模型** |
| CUDA 版本变化（新镜像使用了不同的 CUDA） | 需要重新预热 **Kernel 预编译文件** |
| 仅调整推理引擎参数（如 tp-size） | 不需要重新 Warmup |

**操作方式**：修改 Warmup 的 YAML（如更新 `imagePreload.items` 中的镜像版本），然后重新 apply：

```bash
# 删除旧的 Warmup
kubectl delete rbgwarmup <warmup-name>

# 应用更新后的 Warmup
kubectl apply -f <updated-warmup-yaml>

# 等待完成后再升级推理服务
kubectl get rbgwarmup <warmup-name> -w
```

## 常见问题与排查

### Q: Warmup Pod 状态显示 Failed，如何排查？

```bash
# 查看 Warmup 整体状态
kubectl get rbgwarmup <warmup-name> -o yaml | grep -A10 status

# 查看 Warmup Pod 列表
kubectl get pods -l workloads.x-k8s.io/warmup-name=<warmup-name>

# 查看失败容器的日志
kubectl logs <warmup-pod-name> -c customized-download-model
kubectl logs <warmup-pod-name> -c customized-compile-deepgemm
```

常见失败原因：
+ **模型下载失败**：网络不通、HuggingFace 访问受限（需配置镜像源或代理）、磁盘空间不足
+ **Kernel 编译失败**：GPU 驱动版本不兼容、CUDA 版本不匹配
+ **镜像拉取失败**：镜像仓库认证失败（需配置 `pullSecrets`）

### Q: Warmup Pod 的 `RestartPolicy` 是什么？

Warmup Pod 的 `RestartPolicy` 为 `Never`。这意味着如果 Warmup 容器失败，Pod 不会自动重启。你需要删除失败的 Warmup 资源后重新创建，或排查并修复失败原因后删除失败的 Pod，让 Controller 重新创建。

### Q: 如何在不同节点上预热不同模型？

创建多个 `RoleBasedGroupWarmup` 资源，每个使用不同的 `nodeSelector` 标签。例如：
```yaml
# 节点 A：预热模型 X
nodeSelector:
  model-target: model-x

# 节点 B：预热模型 Y  
nodeSelector:
  model-target: model-y
```

### Q: Warmup 完成后，推理服务 Pod 如何确保调度到已预热的节点？

Warmup 本身**不会**自动设置调度亲和性。你需要在推理服务的 RBG 配置中通过 `nodeAffinity` 使用与 Warmup `nodeSelector` 相同的标签，确保 Pod 调度到已预热的节点。详见上文"部署推理服务"章节的配置示例。

### Q: 节点重启后，预热的数据还在吗？

+ **镜像缓存**：kubelet 管理，通常在节点重启后仍保留（除非磁盘被清理或触发了 image GC）
+ **hostPath 上的模型文件和预编译文件**：持久化在节点本地磁盘，节点重启后仍然存在
+ **page cache**：节点重启后会被清空，模型首次加载需要重新从磁盘读取（但因为模型文件已在本地 hostPath，仍比从远程下载快得多）
