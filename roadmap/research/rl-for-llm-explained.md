# 大模型强化学习（RL for LLM）：What, How, Why

> 面向基础设施工程师的 RL 入门指南
>
> 日期：2026-06-30

---

## 一、What：RL 是什么？

### 1.1 一句话定义

**RL（强化学习）是让大模型从"能说话"进化为"说好话"的训练阶段。**

大模型训练通常分三步：

```
Step 1: 预训练（Pre-training）
  → 从互联网文本学习语言能力
  → 类比：小孩学会说话

Step 2: 监督微调（SFT, Supervised Fine-Tuning）
  → 从人类标注的问答对学习格式和知识
  → 类比：小孩上学，学会回答问题

Step 3: 强化学习（RL）  ← 本文重点
  → 从反馈信号学习什么是"好回答"
  → 类比：小孩在社会中学会什么该说、什么不该说、怎么说更好
```

### 1.2 RL 解决的核心问题

SFT 之后的模型已经能回答问题了，但存在严重问题：

| 问题 | 示例 |
|:---|:---|
| **有害输出** | 教用户制造危险物品、生成歧视性内容 |
| **胡编乱造** | 自信地编造不存在的论文、虚假的历史事件 |
| **质量不稳定** | 同一问题时好时坏，缺乏一致性 |
| **不会拒绝** | 对所有请求都尝试回答，不会说"我不知道" |
| **推理能力弱** | 复杂数学、编程题的推理链不完整 |

**RL 的核心目标**：通过"奖励信号"告诉模型什么是好的回答，让模型自己学会生成更好的内容。

### 1.3 主流 RL 算法

| 算法 | 全称 | 核心思想 | 代表模型 |
|:---:|:---|:---|:---|
| **RLHF** | RL from Human Feedback | 人类标注偏好 → 训练奖励模型 → PPO 优化 | ChatGPT, Claude |
| **PPO** | Proximal Policy Optimization | 经典策略梯度算法，稳定但资源密集 | InstructGPT |
| **GRPO** | Group Relative Policy Optimization | 无需 Critic 模型，组内相对排名作为奖励 | DeepSeek-R1 |
| **DPO** | Direct Preference Optimization | 跳过奖励模型，直接从偏好数据优化 | Llama 3 |
| **REINFORCE** | — | 最简单的策略梯度，变体用于 reasoning | Kimi-K2 |

## 二、How：RL 训练怎么运作？

### 2.1 RLHF/PPO 的完整流程

这是最经典也最复杂的 RL 训练流程。理解了 PPO，其他算法都是简化版。

```
┌─────────────────────────────────────────────────────────────────┐
│                    一轮 RL 训练循环                               │
│                                                                  │
│  ① 生成（Rollout/Generation）                                    │
│     输入一批 prompt → Rollout 引擎生成回答                        │
│     "请解释量子力学" → "量子力学是研究微观粒子的物理学分支..."      │
│                                                                  │
│  ② 评分（Reward Scoring）                                        │
│     Reward 模型对每个回答打分（0-1 分）                            │
│     "量子力学是..." → 0.85 分（好回答）                            │
│     "量子力学就是薛定谔的猫" → 0.3 分（不够好）                    │
│                                                                  │
│  ③ 计算优势（Advantage Computation）                              │
│     用 Critic 模型估计"这个回答比平均水平好多少"                    │
│     Reference 模型计算"回答偏离原始模型多远"（KL 惩罚）             │
│                                                                  │
│  ④ 策略更新（Policy Update）                                      │
│     用 PPO 算法更新 Actor 模型权重                                 │
│     "让模型更倾向于生成高分回答"                                   │
│                                                                  │
│  ⑤ 权重同步（Weight Sync）                                       │
│     将更新后的 Actor 权重同步到 Rollout 引擎                       │
│     → 下一轮生成使用新权重                                        │
│                                                                  │
│  循环 ①→②→③→④→⑤→①→...  直到收敛                                │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 五个角色的职责

| 角色 | 做什么 | 类比 | 资源需求 |
|:---:|:---|:---|:---|
| **Actor** | 被训练的模型，生成回答并接受梯度更新 | 学生 | GPU 密集（训练） |
| **Rollout** | 高速生成引擎（SGLang/vLLM），批量生成回答 | 考试机器 | GPU 密集（推理） |
| **Critic** | 估计每个 token 的价值（Value Function） | 阅卷老师 | GPU 中等（训练） |
| **Reference** | Actor 的冻结副本，防止模型偏离太远（KL 散度） | 标准答案 | GPU 中等（推理） |
| **Reward** | 对生成的回答打分，代替人类评判 | 考官 | GPU 较小（推理） |

### 2.3 为什么需要这么多角色？

直觉上，"训练一个模型"应该只需要一个模型。为什么 RL 需要 4-5 个？

**Actor vs Rollout：为什么要分开？**

Actor 是训练模式（保留梯度、优化器状态），Rollout 是推理模式（纯前向传播，用 KV Cache 加速）。同一个模型在两种模式下的 GPU 内存布局完全不同：

```
训练模式 (Actor):
  模型权重:     30 GB
  梯度:         30 GB
  优化器状态:   60 GB  (Adam: 2x 权重)
  激活值:       20 GB
  总计:        ~140 GB

推理模式 (Rollout):
  模型权重:     30 GB
  KV Cache:     40 GB
  总计:        ~70 GB
```

两者不能高效共存——所以分成两个角色。训练步结束后，Actor 的新权重需要同步到 Rollout。

**Reference：为什么需要冻结副本？**

RL 训练有一个经典问题：**reward hacking**（奖励黑客）。模型学会了"讨好"Reward 模型的技巧，但实际输出变得奇怪。例如：

```
原始模型:  "量子力学是物理学的一个分支，研究微观粒子的行为。"
RL 训练后: "量子力学!!! 非常重要!!! 最好的学科!!! 太棒了!!!"
           → Reward 模型可能给高分（因为情感积极），但回答质量下降
```

Reference 模型用于计算 **KL 散度**（衡量新模型偏离原始模型多远），加一个惩罚项防止模型"跑偏"。

**Critic：为什么不直接用 Reward？**

Reward 只给整个回答一个分数（如 0.85），但 PPO 需要知道**每个 token 的贡献**。Critic（Value Function）估计"从当前 token 开始，后续回答的预期奖励是多少"。这让梯度更新更精准。

注：GRPO 算法跳过了 Critic，用组内相对排名替代，这是 DeepSeek-R1 的重要创新。

### 2.4 数据流示意

```
Prompt 数据集
    │
    ▼
┌──────────┐    生成回答    ┌──────────┐
│ Rollout  │ ─────────────→│  回答集  │
│ (SGLang) │               │ (N 条)   │
└──────────┘               └────┬─────┘
    ▲                           │
    │ ⑤ 权重同步                │ 发送回答
    │                      ┌────▼─────┐
┌───┴──────┐   ④ 梯度更新  │  Reward  │  ② 打分
│  Actor   │ ◄─────────── │  Model   │
│ (训练中) │              └──────────┘
└───┬──────┘                    │
    │                      ┌────▼─────┐
    │   ③ KL 散度           │  Critic  │  ③ 价值估计
    │◄──────────────────── │ (训练中) │
    │                      └──────────┘
    │
┌───▼──────┐
│Reference │  ③ KL 散度基准
│ (冻结)   │
└──────────┘
```

### 2.5 训练-推理循环是最大瓶颈

RL 训练不是连续的梯度下降，而是一个**训练和推理交替的循环**：

```
时间轴:
  |--- 推理阶段 ---|--- 训练阶段 ---|--- 权重同步 ---|--- 推理阶段 ---|
  |   Rollout 忙   |   Actor 忙    |   两者等待     |   Rollout 忙  |
  |   Actor 空闲   |   Rollout 空闲|   ← 瓶颈!     |   Actor 空闲  |
```

**权重同步**是最严重的瓶颈：
- 1T 参数模型（如 Kimi-K2）通过 NCCL 广播需要 **53 秒**
- 通过 RDMA P2P 传输可降至 **7.2 秒**（7.37x 加速）
- 同步期间训练和推理都空闲，GPU 利用率趋近零

这就是为什么编排层（如 RBG）的价值所在——**高效管理推理引擎的暂停/恢复/权重更新生命周期，最小化空闲时间**。

### 2.6 GRPO：简化版 RL

DeepSeek-R1 使用的 GRPO 是 PPO 的重要简化：

```
PPO:   Prompt → Rollout 生成 → Reward 打分 → Critic 估值 → Actor 更新
GRPO:  Prompt → Rollout 生成 N 个回答 → Reward 打分 → 组内排名作为优势 → Actor 更新
                                                       ↑
                                              不需要 Critic 模型!
```

**GRPO 的优势**：
- 少一个模型（不需要 Critic），降低 GPU 需求
- 训练更简单
- 特别适合推理任务（数学、代码），因为可以用规则验证正确性替代 Reward 模型

### 2.7 DPO：更激进的简化

DPO 跳过了 Reward 模型，直接从人类偏好数据优化：

```
RLHF:  人类标注偏好 → 训练 Reward 模型 → 用 Reward 指导 RL 训练
DPO:   人类标注偏好 → 直接优化模型                    ← 跳过 Reward 模型和 RL 循环!
```

**DPO 不需要 Rollout 引擎**——它是纯训练过程。但效果通常不如 PPO/GRPO。

## 三、Why：为什么 RL 重要？

### 3.1 没有 RL 就没有现代 AI 助手

| 模型 | RL 方法 | 关键效果 |
|:---:|:---:|:---|
| **ChatGPT** | RLHF + PPO | 首次展示 RL 让模型"对齐"人类意图 |
| **Claude** | Constitutional AI (RLAIF) | 用 AI 反馈替代人类标注做 RL |
| **Llama 3** | RLHF + DPO | 开源模型首次达到 GPT-4 级别 |
| **DeepSeek-R1** | GRPO | 纯 RL（无 SFT）产生涌现推理能力 |
| **Kimi-K2** | REINFORCE 变体 | 1T MoE 模型的大规模 RL 训练 |
| **Qwen 3** | GRPO + 多阶段 RL | 中文模型的 reasoning 能力突破 |

**DeepSeek-R1 的里程碑意义**：证明了 RL 不仅是"对齐工具"，更是**能力涌现的来源**——模型在 RL 训练中自发学会了思维链、自我纠错、多步推理，而不是从标注数据中学的。

### 3.2 RL 训练的趋势

```
2022-2023:  RL 主要用于安全对齐（防止有害输出）
2024:       RL 开始用于能力提升（推理、代码生成）
2025-2026:  RL 成为核心训练阶段（DeepSeek-R1 证明 RL 可涌现能力）
未来:        RL + 在线学习（持续从用户反馈中学习）
```

### 3.3 RL 训练的资源规模

| 模型 | 参数量 | RL 训练 GPU 需求 | 说明 |
|:---:|:---:|:---:|:---|
| 7B | 7B | 8-16 x A100 | 小规模，单节点可跑 |
| 70B | 70B | 64-128 x A100/H100 | 中等规模，需要多节点 |
| 400B+ MoE | 100B+ 活跃 | 256-1024+ x H100 | 大规模，需要 RDMA 网络 |
| 1T MoE (Kimi-K2) | 64B 活跃 | 1000+ x H100/H200 | 超大规模，权重同步成为瓶颈 |

### 3.4 为什么这与 RBG 相关？

RL 训练的推理侧（Rollout/Reference/Reward）**就是 RBG 已经在做的事情**——部署和管理推理引擎。区别只是生命周期更复杂：

| 维度 | 普通推理服务 | RL 训练中的推理服务 |
|:---:|:---|:---|
| 生命周期 | 启动后持续运行 | 周期性暂停/恢复/权重更新 |
| 权重 | 固定不变 | 每个训练步更新一次 |
| 流量模式 | 持续请求 | 批量生成 → 空闲 → 批量生成 |
| 扩缩容 | 基于请求量 | 基于训练阶段（生成时扩容，训练时缩容） |
| 服务发现 | 地址 + 端口 | 地址 + 端口 + NCCL endpoint + RDMA endpoint |

**RBG 的机会**：成为**连接训练框架和推理引擎的 K8s 原生编排层**——所有主流 RL 框架（Miles/veRL/OpenRLHF）都用 Ray 做编排，没有 K8s Operator。

## 四、图解：PPO 算法核心公式

对于想深入理解的读者，PPO 的核心公式是：

```
L(θ) = E[ min( r(θ) * A,  clip(r(θ), 1-ε, 1+ε) * A ) ]

其中：
  θ     = Actor 模型参数
  r(θ)  = π_θ(a|s) / π_θ_old(a|s)     → 新旧策略的概率比
  A     = Advantage（优势函数）         → 这个动作比平均好多少
  ε     = 裁剪范围（通常 0.1-0.2）      → 限制每步更新幅度
  clip  = 裁剪函数                      → 防止策略变化过大
```

直觉理解：
- `r(θ) * A`：如果一个动作是好的（A>0），增加它的概率；如果是坏的（A<0），降低概率
- `clip`：限制概率变化的幅度，防止一步更新太多导致训练崩溃
- `min`：在"裁剪"和"不裁剪"中取更保守的那个

**GRPO 的区别**：不用 Critic 估计 A，而是：
```
A_i = (R_i - mean(R)) / std(R)    → 在同一 prompt 的 N 个回答中做标准化
```
即"这个回答在这组回答中排第几"作为优势信号。

## 五、总结：基础设施视角的 RL

从基础设施角度，RL 训练的核心挑战是：

```
1. 多角色协调    → RBG 的多角色编排能力
2. 训练-推理交替  → Patio 的 pause/resume/weight-sync API
3. 权重同步瓶颈  → hostNetwork RDMA + 端口模板发现
4. GPU 内存管理  → sleep/wake + 弹性伸缩
5. 大规模部署    → Gang scheduling + 拓扑感知调度
```

**RBG 不需要理解 PPO 算法的数学**，只需要在 K8s 层面高效管理推理引擎的生命周期和服务发现。这就是选项 C 的设计哲学：**让 RBG 做好推理侧编排，把 RL 算法留给训练框架。**
